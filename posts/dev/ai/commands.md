## 코드 리뷰 요청

현재 브랜치와 base 브랜치의 코드를 비교하여 코드리뷰 해줘
코드 리뷰 시, 아래 사항을 고려하여 진행해줘

- 답변 시작하면서 전반적인 변경사항들을 간략하게 요약해줘
- 수정된 코드로 발생 가능한 기존 서비스에서의 사이드 이펙트 들을 점검해줘
  - ex) API 호환성 깨짐, 특정 필드 사라짐 또는 nullable 가능 등
- 성능 이슈가 발생할 수 있거나 코드 비효율 등으로 리팩토링 가능한 부분이 있으면 더 나은 방안을 제안해줘
- 잠재적인 버그가 발생할 수 있는 부분이 있으면 알려줘
- 각 파일간 의존성 정의가 올바른지 확인해줘.
- DRY 원칙에 따라 피드백 해줘
- Dead code 가 있는지도 확인해줘
- 현재 파일의 함수와 메서드들에 타입 힌트가 제대로 작성되어 있는지 확인해줘.

## 커밋 요청

다음 순서로 작업을 수행해줘:

1. **변경사항 확인**

   - `git status --short` 명령으로 현재 변경된 파일 목록 확인
   - 변경사항이 없으면 작업 종료하고 사용자에게 알림

2. **변경사항 분석 및 커밋 메시지 생성**

   - 변경된 파일들의 경로와 내용을 분석
   - 커밋메시지는 영어로 작성
   - 파일 경로, 변경 유형(추가/수정/삭제), 주요 변경 내용을 종합하여 의미있는 커밋 메시지 생성
   - 커밋 메시지 형식: 간결하고 명확하게, 변경사항을 잘 설명하는 형식
   - 예시: "docs: added git worktree guide", "feat: modified commit.sh can get arguments"

3. **커밋 및 푸시 실행**

   - 생성한 커밋 메시지를 `commit.sh` 스크립트의 인자로 전달하여 실행
   - 예: `./commit.sh "생성한 커밋 메시지"`
   - 실행 결과를 확인하고, 에러가 발생하면 사용자에게 알림

4. **결과 확인**
   - 커밋 및 푸시가 성공적으로 완료되었는지 확인
   - 사용자에게 완료 메시지 전달
